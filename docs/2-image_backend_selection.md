# 中国古典连环画生成工具选型分析

本文用于评估本项目在“类似《三国演义》连环画风格的中国古典连环画”场景下，图像生成后端该如何选型。

当前项目已经接入的图像后端包括：RunningHub、Replicate、Zhipu AI，以及新补充的通义万相（DashScope）。

本项目当前已经具备以下后端接入能力：

- `replicate.com`
- `runninghub.cn`
- 预留的 `ComfyUI` 本地后端

同时，LLM 侧已支持 DeepSeek / Kimi / MiniMax 等中文大模型，用于故事生成与提示词构建。

## 一、任务特征

这个项目不是普通的文生图，而是有一组非常具体的约束：

1. 画面必须偏中国传统白描 / 连环画风格。
2. 必须尽量保持黑白线稿、宣纸感、做旧感。
3. 题材是中国古代历史与古典演义，服饰、建筑、兵器要尽量符合历史背景。
4. 需要批量循环生成，不能只追求单张“惊艳”，还要看稳定性和成本。
5. 需要尽量避免彩色、二次元、油画感、现代摄影感。

因此，选型的核心不是“谁最强”，而是“谁最适合这个特定任务”。

## 二、结论先行

如果只给一个现实建议：

- **短期主力：RunningHub.cn**
- **效果上限优先测试：智谱 AI、通义万相**
- **长期终局方案：ComfyUI + 专用 LoRA / 工作流**
- **Replicate.com：适合做备用或国际化补充**

换句话说：

- 如果你要“现在就稳定跑起来”，优先用 **RunningHub**。
- 如果你要“把白描和古风理解做到更好”，优先试 **智谱 AI / 通义万相**。
- 如果你要“未来真正做成自己的白描引擎”，最终还是要落到 **ComfyUI 本地化方案**。

## 四、逐项分析

### 1. Replicate.com

优点：

- 接口成熟，生态丰富。
- 可直接使用 SDXL、FLUX 等模型。
- 适合做通用备选方案。

缺点：

- 对中文古风白描的原生理解一般。
- 想要“连环画线稿感”，通常要靠很强的 prompt 和后处理补足。
- 如果没有专门训练的 LoRA，画风容易偏通用插画或偏彩色风格。

适合场景：

- 备用后端。
- 需要国际平台兼容时使用。
- 后续接入自定义 LoRA 后再提升上限。

### 2. RunningHub.cn

优点：

- 国内接入方便，适合批量生产。
- 中文提示词理解通常比国外通用平台更友好。
- 对当前项目来说，已经是“能直接落地”的方案。

缺点：

- 模型能力取决于其封装的 AI App 或底层渠道。
- 对“极强白描感”“严格连环画版式”的控制力未必是最强。
- 部分任务可能受超时、轮询等工程细节影响。

适合场景：

- 当前项目的默认主力后端。
- 需要兼顾成本、可用性和开发速度时。

### 3. 智谱 AI、通义万相、文心一格、MiniMax 等国产模型

这一类平台的共同特点是：

- 中文理解更自然。
- 对中国古代题材、历史叙事、古风服饰和建筑细节通常更友好。
- 更适合“古典连环画”这种强中文文化语境任务。

其中：

- **智谱 AI**：更适合复杂中文 prompt 和叙事性强的古风图像。
- **通义万相**：古风理解和图像生成综合能力较强，适合作为重点测试对象。
- **文心一格**：中国文化风格感较强，但内容审核可能更严格。
- **MiniMax**：更偏通用中文多模态能力，图像生成并非最强项。

适合场景：

- 作为“效果上限”对照组。
- 作为中文古风题材的重点测试对象。
- 如果后续要做盲测，建议优先把智谱 AI 和通义万相纳入实验。

### 4. ComfyUI 本地方案

优点：

- 最强可控性。
- 可以直接围绕“白描连环画”训练专用 LoRA 或工作流。
- 适合长期沉淀自己的图像生成资产。

缺点：

- 本地部署和工作流维护成本更高。
- 如果机器算力一般，批量生成速度会受影响。
- 需要更多工程投入，短期不如云端 API 省事。

适合场景：

- 作为长期终局方案。
- 适合在云端模型产出足够样本后，回收训练自己的风格模型。

## 五、针对本项目的推荐策略

### 短期

以 **RunningHub.cn** 作为默认主力后端，原因是：

- 已接入，能直接跑。
- 工程改动最少。
- 成本和易用性平衡较好。

### 中期

并行测试 **智谱 AI** 和 **通义万相**：

- 用同一批历史场景做 A/B 测试。
- 对比白描感、历史准确度、人物神态、线条稳定性。
- 如果某个平台明显更适合“古典连环画感”，就把它升级为主力候选。

### 长期

把高质量出图结果沉淀为训练集，逐步迁移到：

- **ComfyUI + 自训练 LoRA**
- 或者 **本地工作流 + 云端 fallback**

这是最适合“做成自己的风格系统”的路线。

## 六、推荐排序

综合“当前可用性、中文理解、古风适配、工程成本、可持续性”五个维度，建议排序如下：

1. **RunningHub.cn**：当前最适合主力落地。
2. **智谱 AI**：最值得优先试验的中文模型。
3. **通义万相**：同样值得重点测试，且已具备官方 DashScope 接口接入条件。
4. **ComfyUI 本地**：长期终局方案。
5. **Replicate.com**：稳定的备用方案。
6. **文心一格 / MiniMax**：可试，但不建议作为第一主力。

## 七、实际建议

如果你现在就要推进项目，我建议的路线是：

1. 继续用 RunningHub 作为默认后端，保证产出。
2. 增加智谱 AI 或通义万相的后端接入，做小批量对比测试。
3. 保留 Replicate 作为 fallback。
4. 中长期把优质样本用于训练专用的白描连环画 LoRA，再切换到 ComfyUI。

这样可以兼顾“马上能用”和“未来可控”两件事。
